# 委任実行

引数: $ARGUMENTS

## 概要

`/plan` と同じ計画フェーズ（Phase 1〜3）を経た後、確認なしでステップコミット自動実行する。

## Phase 1〜3: 計画・承認

`/plan` スキルと同一の手順で計画を作成し、ユーザーの承認を待つ。

## Phase 4: ステップコミット自動実行

承認後、計画の各ステップを実装し**ステップごとに自動コミット**する。ユーザー確認は不要。

### コミットルール

- **1ステップ = 1コミット**（変更を必ずコミット単位で区切る）
- `git add` は変更ファイルを明示的に指定する（`git add -A` 禁止）
- コミット前に `pnpm lint` と `pnpm format:check` を実行する

### コミットメッセージのフォーマット

```text
<type>: <何をしたか（日本語）>

なぜ: <この変更が必要な理由>
根拠: <公式ドキュメントURL、ベストプラクティス、プロジェクト規約への参照>
効果: <この変更によって何が改善されるか>
```

## Phase 5: 実行サマリー

全ステップ完了後、以下を提示する：

```markdown
## 実行サマリー

| #   | コミット | 内容                 | 根拠                    |
| --- | -------- | -------------------- | ----------------------- |
| 1   | abc1234  | ○○を実装             | Playwright 公式: URL    |
| 2   | def5678  | ○○をリファクタリング | Functional POM パターン |

### 何が変わったか

- [変更の概要]

### 設計判断

- [なぜこの実装にしたか]
```
