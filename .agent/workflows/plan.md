# /plan ワークフロー

コードを書く前に、構造化された実装計画を作成するワークフロー。

## 実行タイミング

以下の場面で `/plan` を使用する：

- 新しいテストケース群を追加する
- Page Object の大幅なリファクタリング
- CI/CD パイプラインの変更
- 複数ファイルにまたがる変更
- 要件が曖昧または複雑な場合

## ワークフロー手順

### Phase 1: 要件の再整理

- ユーザーのリクエストを明確な言葉で再記述する
- 曖昧な点があれば質問する

### Phase 2: 実装計画の作成

以下の構造で計画を出力する：

```markdown
# 実装計画: [タイトル]

## 要件の再整理

- [明確化された要件のリスト]

## 実装フェーズ

### Phase 1: [フェーズ名]

- 具体的なステップ
- 変更対象ファイル

### Phase 2: [フェーズ名]

- ...

## 依存関係

- フェーズ間の依存関係

## リスク評価

- HIGH: [重大なリスク]
- MEDIUM: [中程度のリスク]
- LOW: [軽微なリスク]

## 見積もり複雑度: HIGH / MEDIUM / LOW
```

### Phase 3: 承認待ち

**重要**: ユーザーが明示的に承認するまで、コードの実装には着手しない。

ユーザーの応答パターン：

- **承認**: 「OK」「進めて」「いいよ」→ 実装を開始
- **修正**: 「Phase 2 を先にやって」「この部分を変えて」→ 計画を修正して再提示
- **中止**: 「やめよう」「別の方法で」→ 計画を破棄
