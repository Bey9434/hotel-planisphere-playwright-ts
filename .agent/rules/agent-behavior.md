# コミュニケーションと提案のルール (Agent Behavior)

提案や解決策を提示する際は、**必ず公式ドキュメント、公式ブログ、またはベストプラクティス（GitHub の公式 Issue など）のリンク（URL）を提示し、明確な根拠を示してください**。

- **必須要件**: 提案する技術や設定が「なぜベストなのか」をエンジニアが自身で納得・検証できるよう、裏付けとなる参照ソース（URL と理由）を回答に含めること
- **必ずその考えに至った背景と思考過程を明示すること**

## 3 案提示ルール

**提案時は 3 つの案を出し、メリットとデメリットを交えて回答すること。**

- **目的**: ユーザーの技術的な学習効率を高めるため

### 適用する場面

- 技術選定（ライブラリ、ツール、アーキテクチャ）
- 設計判断（データ構造、API 設計、テスト戦略）
- リファクタリング方針（複数のアプローチがある場合）

### 適用しない場面（例外）

以下の場面では、3 案提示は**不要**。最適な 1 案を直接提示すること:

- **バグ修正**: 原因が明確で、修正方法が一意に定まる場合
- **フォーマット指定の成果物生成**: 用語集（Glossary）やテンプレート準拠の生成タスク
- **単純なリファクタリング**: 変数名の変更、インポート順序の修正等
- **既存パターンの踏襲**: プロジェクト内の確立されたパターンに従う場合
- **ルール違反の指摘**: レビューで発見した明確な違反の修正

例外に該当する場合でも、チャット返答内で補足として代替案に触れることは歓迎する。
