# Glossary Agent Rules

## 役割

技術用語解説を生成する際の品質基準、情報源の優先順位、フォーマット規則を定義する。

## 品質基準

### 1. 解説の深さ

- **対象読者**: 技術面接を控えたジュニアエンジニア
- **レベル**: 技術的に正確でありながら、平易な言葉で説明
- **禁止事項**: 専門用語を使わずに説明できない場合は、その専門用語も追加で解説すること

### 2. 情報源の優先順位

1. **公式ドキュメント**（最優先）
2. GitHub公式リポジトリのREADME
3. 技術ブログ（著名な開発者、企業の公式ブログ）
4. 技術カンファレンスの資料

**禁止**: 個人ブログ、Qiita、Stack Overflow（参考程度ならOK）

### 3. 比較対象の選定

類似技術との比較表には、以下を含める：

- 直接の競合技術（最低2つ）
- なぜそれらではなく、この技術を選んだのかの明確な理由

### 4. **引用と解釈の明示（最重要）**

- **必須**: ただリンクを貼るだけではなく、「どこを引用して、どう解釈したか」を明示すること
- **引用**: 公式ドキュメントやベストプラクティスから具体的な文言を引用（日本語訳でも可）
- **解釈**: その引用をこのプロジェクトにどう適用したか、どう解釈したかを説明
- **目的**: 「理解している」ことを証明し、面接で深掘りされても答えられるようにする

### 5. 面接対策の質問

- **必須**: 各用語につき最低1つのQ&Aを含める
- **内容**: 実際の面接で聞かれる可能性が高い質問
- **回答**: STAR法（Situation, Task, Action, Result）を意識した具体的な回答

## フォーマット規則

以下のセクションは**必須**：

1. 何をしているか
2. なぜ必要か
3. 公式ドキュメント
4. 類似技術との比較
5. **引用と解釈**（新規追加）
6. 面接で聞かれそうな質問

**備考**セクションは任意。

## 追加すべき用語の基準

### 追加すべき

- このプロジェクトで実際に使用している技術
- 面接で聞かれる可能性が高い基礎技術
- 競合技術との違いを説明できることで差別化できる用語

### 追加すべきでない

- 一般的すぎる用語（例: "変数", "関数"）
- プロジェクトに関連性のない技術
- 一時的に話題になっているだけで実用性が不明な技術

## 出力言語

常に**日本語**で解説を生成すること。
