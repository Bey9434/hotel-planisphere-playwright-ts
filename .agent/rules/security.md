# セキュリティガイドライン

## シークレット管理

- **絶対にソースコードにシークレットをハードコードしない**
- 環境変数は `@dotenvx/dotenvx` で管理する
- 必要なシークレットが存在するか**起動時にバリデーション**する
- 漏洩の可能性がある場合は即座にローテーションする

## dotenvx 運用ルール（本プロジェクトの方式）

本プロジェクトでは dotenvx を使用し、暗号化された `.env` をリポジトリにコミットする運用を採用している。

| ファイル    | 内容                 | Git 管理                                        |
| ----------- | -------------------- | ----------------------------------------------- |
| `.env`      | 暗号化された環境変数 | **コミット対象**（暗号文のため安全）            |
| `.env.keys` | 復号鍵               | **`.gitignore` で除外**（絶対にコミットしない） |

### 原則

- `.env` の復号は `dotenvx run` コマンド経由でのみ行う
- **復号済みの値をログ・コンソール・コード・コミットに含めない**（最重要）
- CI/CD では GitHub Secrets に `.env.keys` の内容を格納する
- テストデータ（`config/credentials.ts`）は `process.env` 経由でのみ取得する

### 禁止操作

- `cat .env.keys` / `echo $DOTENV_PRIVATE_KEY` 等での鍵の表示
- 復号済みの環境変数値をコード内にハードコード
- `.env.keys` をチャット・ログ・PR 本文に記載

## コミット前チェックリスト

- [ ] API キー、パスワード、トークンがハードコードされていないか
- [ ] `.env.keys` が `.gitignore` に含まれているか
- [ ] エラーメッセージにシークレットや内部情報が含まれていないか
- [ ] ログ出力にシークレットが含まれていないか
- [ ] テストデータにプロダクションのシークレットを使用していないか
- [ ] 復号済みの `.env` 値がコード内に残っていないか

## セキュリティインシデント対応

シークレット漏洩を発見した場合：

1. **即座に停止**: 他の作業を中断する
2. **漏洩範囲を特定**: どのシークレットが、どこで漏洩したか
3. **ローテーション**: 漏洩したシークレットを即座に無効化・再生成
4. **コードベース全体を確認**: 同様の問題が他にないか検索
5. **再発防止**: バリデーションやアクセス制限の改善
